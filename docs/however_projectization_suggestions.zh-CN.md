# pipecat 私有化改造建议清单（60 条）

> 目标：将通用框架仓库改造为可持续维护、可部署、可回归的个人工程发行版。

## A. 品牌与定位（1-10）

1. 在 README 首页统一品牌为 `howeverpipecat`，保留 upstream 来源说明。  
2. 增加自有 Logo 资产并在 README 头部展示。  
3. 使用一句话定义仓库定位：实时语音智能体工程发行版。  
4. 明确“上游框架能力”和“fork 增量能力”的边界。  
5. 固化 GitHub Topics，避免每次手工维护。  
6. 增加仓库描述设置脚本，支持一键更新元信息。  
7. 将部署导向文案放在前 1 屏，减少新手路径成本。  
8. 增加“系列仓库地图”说明与其它项目关系。  
9. 制定统一版本命名规则（如 `vX.Y.Z-however.N`）。  
10. 统一文档语言策略：中文为主，代码和关键词保留英文。  

## B. 包名与命名空间治理（11-20）

11. 发布包名改为 `howeverpipecat-ai`，与 upstream 包名隔离。  
12. 保留 `pipecat` import 命名空间，避免破坏生态兼容。  
13. 逐步引入 `OllamaLLMService` 命名规范，保留旧名兼容别名。  
14. 为兼容别名提供 deprecation warning 与迁移窗口。  
15. 给所有 fork 新增模块加 `however_` 前缀，降低冲突概率。  
16. 给新增脚本统一命名风格（`however_*`）。  
17. 补一份命名规范文档，明确类名、文件名、环境变量前缀。  
18. 增加命名 lint（拼写、大小写、缩写一致性）。  
19. 将环境变量前缀统一为 `HOWEVER_`。  
20. 在 changelog 中单列“命名变更”章节。  

## C. 配置与敏感信息治理（21-30）

21. 增加 `env.however.example`，隔离 fork 运行时变量模板。  
22. 增加 `configs/services.example.yaml`，统一服务拓扑配置入口。  
23. 将 DB/Redis/Ollama 默认地址改为内网占位域名。  
24. 禁止在代码中硬编码账号、密码、token。  
25. 增加本地健康检查脚本，验证依赖可达性。  
26. 在 CI 中加入敏感信息扫描。  
27. 增加变量最小校验（URL scheme、host、timeout 范围）。  
28. 配置项支持 env 优先，模板文件兜底。  
29. 增加 `.env.however.local` 私有文件约定与忽略规则。  
30. 增加配置变更审计清单（新增、删减、影响模块）。  

## D. 依赖与构建治理（31-40）

31. 定义“核心依赖最小集合”和“扩展依赖集合”边界。  
32. 为第三方服务 extra 增加稳定性评级（stable/beta/exp）。  
33. 对高频更新依赖建立月度升级策略。  
34. 建立依赖升级前后的回归基线。  
35. 将自引用 extras 全量替换为新包名，避免发布后失效。  
36. 为 Docker/本地开发分别维护依赖建议。  
37. 补齐 Python 3.11/3.12 差异兼容说明。  
38. 在 release 前跑依赖安全扫描。  
39. 增加依赖冲突 FAQ（典型冲突与解法）。  
40. 对音频栈相关依赖提供系统包安装说明。  

## E. 架构与代码层改造（41-50）

41. 抽离 fork 运行时配置加载模块，避免脚本散落解析。  
42. 将健康检查脚本逻辑下沉到可测试的 `src` 模块。  
43. 给 Ollama 服务接入环境变量 base_url 解析。  
44. 增加兼容类的单元测试，确保迁移期稳定。  
45. 为关键服务类补充行为契约测试。  
46. 增加“离线模式”支持，便于本地 CI 与演示。  
47. 对示例代码统一改为可配置 base_url。  
48. 统一日志字段（env、region、latency、status）。  
49. 新增运行时故障分类（配置错、网络错、认证错）。  
50. 为新增模块补 docstring 和类型标注。  

## F. 文档、部署与运营（51-60）

51. 重写 README：用途、改动点、部署方式、差异化四段式。  
52. 新增“与 upstream 差异”章节，按模块分组。  
53. 新增“快速迁移指南”章节（上游 -> fork）。  
54. 在 README 放置最短可运行路径（5 分钟内可跑）。  
55. 将常见错误排查写成可执行 checklist。  
56. 增加 `LICENSE.HOWEVER` 说明 fork 归属与分发边界。  
57. 加入仓库元信息脚本（描述、主页、topics）。  
58. 建立 issue 模板：复现步骤 + 环境信息 + 日志片段。  
59. 建立 release note 模板：功能、风险、回滚方案。  
60. 增加 90 天演进路线图（稳定性、性能、生态兼容、部署）。
